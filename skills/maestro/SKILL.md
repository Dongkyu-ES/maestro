# maestro — task-aware orchestration

작업 하나를 받아서 (1) **분류**하고 → (2) 맞는 **패턴**을 고르고 → (3) 역할별 CLI를 **조합**하고 → (4) 역할마다 **필요한 컨텍스트만 적재**(필요 시 워크트리 분리 + 선택적 MCP 주입)하고 → (5) 오케스트레이션을 **실행**해 하나의 결과로 **종합**한다.

## 단 하나의 철칙 (이게 깨지면 maestro가 다시 무거워진다)

**maestro는 이미 있는 도구를 *조합*만 한다. 새 인프라를 만들지 않는다.**
- 무거운 일(워크트리 분리·선택 주입·DAG 팬아웃)은 이 repo의 `maestro` CLI 엔진에 위임한다.
- 읽기 전용 일(패널·스카우트·검증·리뷰)은 인세션 서브에이전트로 가볍게 돌린다.
- acceptance contract도, ledger를 척추로 삼는 것도 **기본 경로엔 없다**. 증거/게이트는 `--prove`/`--gate`로 명시할 때만. 가벼운 작업은 가볍게 끝나야 한다.

## 워크플로우

### 1. 분류 → 패턴 선택 (하나만; 사용자가 지정하면 그걸 따른다)

| 신호 | 패턴 | 왜 |
|------|------|-----|
| 산출물 1개, 품질이 중요, "제대로 됐는지 확신 필요" | **build → 독립 리뷰 루프** | 만든 모델과 *다른* 모델이 리뷰 → 자기검증의 맹점을 깬다 |
| 같은 목표에 독립 시도 여러 개, "여러 안 중 best" | **팬아웃 + 선택** | N개 병렬 시도 → 심판이 선택/종합 |
| 결정/설계 질문, 파일 변경 없음, "어느 쪽이 맞아?" | **패널 / 디베이트** | 서로 다른 모델·관점의 조언자 → 종합 |
| 넓고 모호한 빌드, "어디부터 손대야 할지 모름" | **scout → plan → build → verify** | 탐색→계획→구현→검증 파이프라인 |

각 패턴의 구체 실행 레시피(명령·역할→CLI·워크트리 여부·종합 단계)는 **`references/patterns.md`** 참조.

**기본은 단일 경로.** fan-out·panel(다중 에이전트)은 비싸다 — 연산을 정규화하면 단일 에이전트가 다중 디베이트와 동급이거나 우위이고, 이득은 3~5에서 포화하며, 디베이트는 종종 숙의가 아니라 앙상블이라 다수압력이 옳은 소수의견을 누른다. 그러니 **다중화는 예외**로, 다음 신호 중 하나 이상일 때만: (a)진짜 독립적인 시도가 가치 있음, (b)단일 시도의 확신이 낮음, (c)어렵거나 진짜 agentic한 작업. 쓸 때도 최대 3~5, 그리고 *수*보다 **관점 다양성**(다른 모델/렌즈)을 우선한다. 규칙을 더 못 늘려서가 아니라 — 추가하는 모든 규칙은 "결과를 개선하는 가장 가벼운 기존 경로를 고른다"는 단 하나의 규칙과 경쟁한다.

### 2. 역할 → CLI 매핑

기본 휴리스틱(빌드=codex workspace-write, 리뷰=다른 모델인 codex gpt-5.5 high 또는 claude, 스카우트/심판=claude·Explore)과 각 CLI의 정확한 헤드리스 호출법(특히 codex의 `</dev/null` 필수 함정)은 **`references/cli-map.md`** 참조. 사용자가 역할별 CLI를 지정하면 그걸 우선한다.

### 3. 조건부 프로비저닝 ("매직" = 필요한 것만 적재)

**핵심 규칙: 워크트리/주입은 *파일을 병렬로 바꾸는 단계*에만 쓴다.** 읽기 전용 단계는 인세션으로 — 워크트리도, 주입도 없음.

- **파일 변경 / 병렬 빌드 단계** → `maestro magic run "<goal>" --executor <cli>` 에 위임 (이미 detect→resolve→최소 `.mcp.json` 주입→스폰→정리를 다 함), 또는 여러 노드면 `graph.json`을 써서 `maestro orchestrate run --file graph.json`.
- **읽기 전용 단계**(패널·스카우트·검증·리뷰) → 인세션 Task 서브에이전트 또는 직접 `codex exec --sandbox read-only`. **워크트리 만들지 말 것.**

언제 워크트리를 쪼개고 언제 인세션으로 갈지, `magic`/`orchestrate` 명령 치트시트는 **`references/provisioning.md`** 참조.

### 4. 실행 + 5. 종합

역할별 산출물은 작업 디렉토리 하위 `_workspace/`에 파일로 모은다 (`{단계}_{역할}_{산출물}.{ext}`). 최종적으로 하나의 결과로 종합해 사용자에게 보고한다. 팬아웃 후엔 `maestro worktrees cleanup`으로 잔여 워크트리를 정리한다.

**핸드오프는 최소 구조화로.** 다음 역할에 넘길 땐 전체 이력을 복붙하지 말고 4필드면 충분: **목표 / 입력 ref(경로·이전 산출물) / 제약 / 산출물 경로** (대략 200~500토큰). 전체 대화 전달은 토큰만 먹고 컨텍스트를 흐린다.

## 증거는 옵트인 (기본 OFF)

기본 실행은 ledger/prove 산출물을 만들지 않는다. 사용자가:
- `--prove` 를 요청하면 → `maestro magic run ... --prove` (카나리 소비 증명)
- `--gate <cmd>` 를 요청하면 → 그 검증 명령을 깨끗한 체크아웃에서 재실행하는 게이트를 건다

기존 ledger/prove 코드와 테스트는 그대로 살아있다 — 단지 기본 경로에서 내렸을 뿐. 사용자가 "증거", "검증된 완료", "audit" 등을 명시할 때만 켠다.

## 무엇을 호출하나 (만들지 말고 부른다)

| 필요 | 호출 |
|------|------|
| detect+resolve dry-run | `maestro magic plan "<goal>"` |
| 워크트리+선택주입+스폰+정리 | `maestro magic run "<goal>" --executor codex\|claude\|agy [--prove]` |
| DAG 팬아웃 (노드 `{id, goal, deps?, executor?}`) | `maestro orchestrate run --file graph.json [--concurrency N]` |
| 워크트리 정리 | `maestro worktrees cleanup` |
| 읽기 전용 조언자/리뷰어 | 인세션 `codex exec -m gpt-5.5 -c model_reasoning_effort="high" --sandbox read-only "<p>" </dev/null` 또는 Task(Explore/general-purpose) |

## 트리거하지 말 것

한 역할이 순차로 끝낼 단순 작업("이 함수 고쳐", "오타 수정")엔 maestro를 쓰지 말고 직접 처리한다. maestro의 가치는 *여러 역할의 협업*이 결과를 실제로 개선할 때만 나온다. 오버헤드(워크트리·여러 CLI 스폰)가 이득을 넘으면 쓰지 않는 게 맞다.
